# 日記を書く

プロジェクトルートの `日記.md` に、今日の作業を記録する。ファイルが無ければ以下の見出しで新規作成してから追記する。

```markdown
# komichi 開発日記

日々の作業内容・詰まった点や注意事項・次にやることを記録する。`/diary` スキルで追記される。
```

## 手順

1. **今日の日付を確認する。** `date +%Y-%m-%d` 等で取得する（ユーザーの申告やシステムリマインダーの日付を優先し、憶測しない）。
2. **今日の会話・作業内容を振り返る。** このセッション（および会話が圧縮されている場合はサマリ）で実際に行ったやり取りを元に書く。裏付けが薄い場合は `git log --oneline --since="today 00:00"` や `git status` も参考にしてよいが、コミットログだけでは分からない「試行錯誤の過程」こそがこのスキルの価値なので、会話内容を優先する。
3. **以下の3見出しで追記する。** `日記.md` の末尾に新しい日付セクションを追加する(Edit使用。既存内容は残す)。

   ```markdown
   ## YYYY-MM-DD

   ### 今日行ったこと
   - (具体的に。ファイル名・機能名・コマンド名を含める)

   ### 詰まった事・注意事項
   - (原因が非自明だったもの、次に同じ罠を踏まないための具体的な教訓。「〜がうまくいかなかった」で終わらせず、原因と解決法/回避策までセットで書く)

   ### 次に行う事
   - (未完了のタスク、ユーザーの返答待ちの事項、次にやると決まっていること)
   ```

4. 同じ日付のセクションが既に存在する場合は、新規セクションを作らず既存セクションに追記する（1日1セクション）。
5. **書いたら即座にコミット・pushする。** 確認は不要（このスキルの標準動作として毎回自動で行う）。

   ```bash
   git add 日記.md
   git commit -m "日記: YYYY-MM-DD"
   git push
   ```

   - `日記.md` 以外に変更が残っている場合でも、コミットは `日記.md` だけを対象にする（他の未コミット変更を巻き込まない）。
   - `git push` が `[rejected] (fetch first)` 等で失敗した場合は `git pull --rebase` してから再度 `git push` する。それでも失敗する場合（認証エラー等）はユーザーに報告する。
6. 書き終えたら、追記した内容とpush結果を短く要約してユーザーに報告する（ファイル全文を貼り直さない）。

## 注意

- 「詰まった事」は当たり障りのない一般論ではなく、実際に発生した具体的なエラーメッセージ・原因・解決手順を書く（このプロジェクトの過去の日記やコミットメッセージの書きぶりに合わせる: 具体的で簡潔、背景の"なぜ"を一言添える）。
- 個人的・機密性の高い情報（トークン・パスワードそのもの、他者のプライバシーに関わる内容）は書かない。
- ユーザーから明示的に「今日は◯◯をやった」等の追加情報があれば、それを優先して反映する。
